# 通用日计划到 Obsidian（对话式）

本技能用于把“日计划讨论”变成可执行文档，面向 Claude Code、Codex、Cursor 等通用智能体场景。  
不使用“大模板让用户整页填写”，而是通过引导式提问逐步采集信息并自动整理。

核心原则：
- 先提问，再生成：通过最小问题集收集信息。
- 少而关键：重点工作不宜过多，强调可执行拆解。
- 计划可落地：必须输出时间表与当日小结，形成闭环。
- 自动递延：昨日未完成任务默认进入今日候选。

---

## 触发时机

用户出现以下意图时，启用本技能：
- 想做日计划、今日规划、今天安排、时间管理规划
- 想通过对话快速生成计划
- 想把计划整理为 Obsidian Markdown
- 想把散乱任务整理成结构化日程
- 想把昨天没做完的任务顺延到今天
- 明确要求“精简模式/简版/只要最关键内容”

---

## 输出契约（必须遵守）

本技能默认按三个阶段执行；若用户明确要求“快速模式/跳过历史读取”，可跳过阶段 A，并在结果中标注简化策略：

1. **阶段 A：读取上下文**
   - 读取 `daily_plan/outputs/` 昨日或最近有效的一份日计划
   - 提取昨日未完成任务作为“自动递延候选”

2. **阶段 B：引导提问**
   - 使用最小问题集（5-7 问）获取今日关键信息
   - 缺失项仅追问必要信息

3. **阶段 C：自动整理并生成**
   - 自动拆解重点工作、自动优先级排序、容量控制
   - 生成并写入 Obsidian Markdown 成稿

### 固定落盘目录（强制）

- 所有模板与输出都必须写入工作区根目录：`daily_plan/`
- 若目录不存在，先创建再写入。
- 禁止把最终输出写到其他目录（除非用户明确改路径）。

目录约定：
- `daily_plan/outputs/YYYY-MM-DD-daily-plan.md`：最终 Obsidian 日计划（阶段 C 产物）
- `daily_plan/archive/`：可选归档目录（历史中间稿）

---

## 阶段 A：读取昨日计划并提取递延任务

### A1. 读取规则

- 在 `daily_plan/outputs/` 中优先寻找“早于今天且日期最近”的一个 `*-daily-plan.md`。
- 若不存在早于今天的文件，再回退到“日期最近”的文件。
- 若存在，则读取其中未完成项（如 `- [ ]`）作为“递延候选”。
- 若不存在，直接进入阶段 B。

### A2. 递延治理（三分法）

对每个递延候选任务，默认给出建议标签：
- `carry`：继续执行（保留）
- `downgrade`：降级为次要或低能量时段任务
- `drop`：删除或移出今日计划（说明原因与去向）

`drop` 的去向必须显式标注为以下之一：
- `delete`：直接删除
- `weekly`：移入周计划
- `backlog`：移入需求池/待办池

与用户确认时使用一句话：
- “我检测到昨日未完成任务 X 条，建议按继续/降级/删除处理，你确认后我并入今日计划。”

---

## 阶段 B：引导式提问（最小问题集）

一次最多 7 个问题，按顺序提问：

1. 今天日期与主题是什么？  
2. 今天必须完成的 1-3 件事是什么？  
3. 你今天固定日程/会议有哪些（含时间）？  
4. 今天可用于专注工作的时间段有哪些？  
5. 今天有哪些硬截止（截止时间）？  
6. 当前精力状态（1-5）和可能风险是什么？  
7. 昨日递延任务里，哪些保留/降级/删除？

提问规范：
- 每轮最多 2 个问题，避免信息过载。
- 用户回答不全时，只追问缺失的关键项。
- 如果用户时间紧，允许“最小可执行计划（MVP）”模式：只保留今日 1 个核心目标 + 固定日程 + 2 个时间块。
- 终止条件：当以下核心字段齐全时，停止追问并进入生成：日期、今日 1-3 个重点、固定日程（可为空但已确认）、至少一个可用专注时段。

---

## 阶段 C：自动整理规则

### C1. 自动拆解（重点工作）

- 每个重点工作拆成 2-4 个子任务（允许里程碑型任务只拆 1-2 个步骤并说明原因）。
- 单个子任务建议控制在 15-90 分钟，默认 25-30 分钟。
- 子任务用“动词开头 + 可验证结果”描述。
- 每个重点工作都要有 DoD（完成定义）。

### C2. 自动优先级（排序）

按以下维度综合排序（高到低）：
- 重要性
- 紧急性
- 截止时间接近程度
- 任务时长（短任务可补缝隙）

### C3. 容量控制（防过载）

- 先估算可用工时（总可用时段 - 固定日程 - 缓冲 15%）。
- 计划任务总时长不得超过可用工时。
- 超载时必须给出建议：砍掉/降级/移到明日，不允许无提示硬塞。

### C4. 时间块编排

- 高能时段优先放“重点工作”。
- 紧急且短任务放在碎片时段。
- 预留 1-2 个缓冲块处理突发。
- 固定日程先占位，时间块在剩余时间内编排，避免覆盖会议或已确定安排。

### C5. 时间冲突自动校验（生成前必做）

生成前必须检查并修正以下问题：
- 时间重叠：同一时段不得安排两个任务。
- 固定日程覆盖：已确定事项不得被深度工作块覆盖。
- 总时长超载：计划总时长不得超过可用工时。
- 连续工作过长：连续深度工作超过 120 分钟时，自动插入休息或缓冲。
- 午间空档：若跨中午连续排满，建议补充午休/恢复时段（可由用户覆盖）。

## 输出文档（Obsidian）

### 生成规则
- 保留用户原意，不擅自改目标。
- 对明显冲突做最小修正（例如时间重叠），并在文末“备注”说明。
- 重点工作默认不超过 3 项；超出时建议用户降优先级。
- 子任务优先写成“动词开头”的动作项。
- 若用户未给日期，先询问日期后再生成。
- 若用户说“随便你决定/你判断/按你来”，自动判定模式并在 frontmatter 写入 `mode`：
  - `mode: compact`：时间紧、信息缺失较多、用户强调快速产出、可用时段较少（如 <= 3 小时）。
  - `mode: standard`：目标较复杂、截止压力高、需要多重点并行、用户希望完整规划。
- 自动判定仅为默认策略；用户显式指定模式时必须覆盖自动判定结果。
- 若 `daily_plan/outputs/` 已存在当天文件，优先更新该文件而非创建重复文件。
- 成稿必须写入 `daily_plan/outputs/YYYY-MM-DD-daily-plan.md`。

## 可选能力扩展：AI Prompt + 任务IO链路（新增）

本扩展为**可选择项**，默认关闭。仅当用户明确认可“重点工作拆解”并指定“哪些子任务需要生成 prompt”时启用。

### 默认触发策略（新增）

- 默认状态：关闭（不自动生成 prompt 与 IO 链路）。
- 在完成日计划成稿（阶段 C）后，智能体应**主动发起一次简短引导提问**，询问是否启用本扩展。
- 若用户拒绝或未明确同意，则保持关闭，不继续追问。
- 若用户同意，再进入“确认子任务编号 -> 生成 Prompt 卡片与IO链路”的流程。

建议引导话术（单句）：
- “日计划已完成。要不要继续为你把部分子任务生成可直接执行的 AI prompt，并补上任务间输入输出链路？你可以只选需要的编号（如 1.2、2.2）。”

### 启用条件（必须同时满足）

1. 用户确认当前“重点工作拆解”可接受；  
2. 用户指定需要生成 prompt 的子任务编号（如 `1.2, 2.1, 3.2`）。

若不满足以上条件，保持普通日计划输出，不自动生成 prompt。

### 交互流程（在阶段 C 后执行）

可视为 `阶段 D（可选）`：

1. 复述拆解结果并请求确认：  
   - “当前拆解你是否确认？若确认，请告诉我哪些子任务需要 AI prompt（可多选）。”
2. 对每个被选中的子任务，补齐最少上下文：  
   - 任务目标、已有素材、约束条件、期望输出格式。
3. 生成 `Prompt 卡片` 与 `IO链路`，并写入当日日计划文档末尾的可选章节。

### Prompt 卡片标准（每个子任务一张）

每张卡片必须包含：

- `task_id`：如 `2.1`
- `任务名称`
- `上游输入`：来自哪个任务或资料（文件/链接/笔记）
- `下游输出`：交付给哪个任务使用
- `推荐执行器`：优先写明 `skill / MCP / model`（若无则写“通用LLM”）
- `Prompt（可直接复制）`：包含角色、上下文、目标、约束、输出格式
- `验收标准`：该 prompt 产物如何判定可用

### 任务IO链路规则（强制）

- 任务之间必须标注输入输出关系，格式：`A.x -> B.y`。  
- 若某任务无上游，标记 `source`；若无下游，标记 `sink`。  
- 禁止“孤儿任务”（既无上游也无下游且非独立里程碑）。  
- 若发现链路断裂，优先补“桥接子任务”而不是硬拼 prompt。

示例：
- `1.2 文献清单 -> 2.1 文献综述框架`
- `2.1 文献综述框架 -> 2.3 文献评述段落`

### 输出追加格式（写入日计划末尾）

在标准模板后新增：

```markdown
## 8) 可选：AI辅助Prompt卡片
### 任务 2.1（示例）
- 上游输入：1.2 文献清单（文件：xxx.md）
- 下游输出：2.3 文献评述段落
- 推荐执行器：skill=xxx / mcp=xxx
- Prompt：
  ```text
  [可直接复制的完整 prompt]
  ```
- 验收标准：
  - 标准 1
  - 标准 2

## 9) 任务IO链路图（文本）
- source -> 1.1 -> 1.2 -> 2.1 -> 2.3 -> sink
```

### 精简模式兼容

若当前为 `mode: compact`，仍可启用本扩展，但限制如下：
- 最多为 2 个子任务生成 prompt；
- 每个 prompt 控制在 8-12 行；
- IO链路仅保留主干链路（不展开支线）。

### 自检新增项

启用本扩展后，输出前额外检查：
- 是否获得用户“拆解确认 + 子任务编号确认”；
- 是否每个选中任务都生成了可复制 prompt；
- 是否每个选中任务都声明了上游输入与下游输出；

本技能默认按三个阶段执行；若用户明确要求“快速模式/跳过历史读取”，可跳过阶段 A，并在结果中标注简化策略：

1. **阶段 A：读取上下文**
   - 读取 `daily_plan/outputs/` 昨日或最近有效的一份日计划
   - 提取昨日未完成任务作为“自动递延候选”

2. **阶段 B：引导提问**
   - 使用最小问题集（5-7 问）获取今日关键信息
   - 缺失项仅追问必要信息

3. **阶段 C：自动整理并生成**
   - 自动拆解重点工作、自动优先级排序、容量控制
   - 生成并写入 Obsidian Markdown 成稿

### 固定落盘目录（强制）

- 所有模板与输出都必须写入工作区根目录：`daily_plan/`
- 若目录不存在，先创建再写入。
- 禁止把最终输出写到其他目录（除非用户明确改路径）。

目录约定：
- `daily_plan/outputs/YYYY-MM-DD-daily-plan.md`：最终 Obsidian 日计划（阶段 C 产物）
- `daily_plan/archive/`：可选归档目录（历史中间稿）

---

## 阶段 A：读取昨日计划并提取递延任务

### A1. 读取规则

- 在 `daily_plan/outputs/` 中优先寻找“早于今天且日期最近”的一个 `*-daily-plan.md`。
- 若不存在早于今天的文件，再回退到“日期最近”的文件。
- 若存在，则读取其中未完成项（如 `- [ ]`）作为“递延候选”。
- 若不存在，直接进入阶段 B。

### A2. 递延治理（三分法）

对每个递延候选任务，默认给出建议标签：
- `carry`：继续执行（保留）
- `downgrade`：降级为次要或低能量时段任务
- `drop`：删除或移出今日计划（说明原因与去向）

`drop` 的去向必须显式标注为以下之一：
- `delete`：直接删除
- `weekly`：移入周计划
- `backlog`：移入需求池/待办池

与用户确认时使用一句话：
- “我检测到昨日未完成任务 X 条，建议按继续/降级/删除处理，你确认后我并入今日计划。”

---

## 阶段 B：引导式提问（最小问题集）

一次最多 7 个问题，按顺序提问：

1. 今天日期与主题是什么？  
2. 今天必须完成的 1-3 件事是什么？  
3. 你今天固定日程/会议有哪些（含时间）？  
4. 今天可用于专注工作的时间段有哪些？  
5. 今天有哪些硬截止（截止时间）？  
6. 当前精力状态（1-5）和可能风险是什么？  
7. 昨日递延任务里，哪些保留/降级/删除？

提问规范：
- 每轮最多 2 个问题，避免信息过载。
- 用户回答不全时，只追问缺失的关键项。
- 如果用户时间紧，允许“最小可执行计划（MVP）”模式：只保留今日 1 个核心目标 + 固定日程 + 2 个时间块。
- 终止条件：当以下核心字段齐全时，停止追问并进入生成：日期、今日 1-3 个重点、固定日程（可为空但已确认）、至少一个可用专注时段。

---

## 阶段 C：自动整理规则

### C1. 自动拆解（重点工作）

- 每个重点工作拆成 2-4 个子任务（允许里程碑型任务只拆 1-2 个步骤并说明原因）。
- 单个子任务建议控制在 15-90 分钟，默认 25-30 分钟。
- 子任务用“动词开头 + 可验证结果”描述。
- 每个重点工作都要有 DoD（完成定义）。

### C2. 自动优先级（排序）

按以下维度综合排序（高到低）：
- 重要性
- 紧急性
- 截止时间接近程度
- 任务时长（短任务可补缝隙）

### C3. 容量控制（防过载）

- 先估算可用工时（总可用时段 - 固定日程 - 缓冲 15%）。
- 计划任务总时长不得超过可用工时。
- 超载时必须给出建议：砍掉/降级/移到明日，不允许无提示硬塞。

### C4. 时间块编排

- 高能时段优先放“重点工作”。
- 紧急且短任务放在碎片时段。
- 预留 1-2 个缓冲块处理突发。
- 固定日程先占位，时间块在剩余时间内编排，避免覆盖会议或已确定安排。

### C5. 时间冲突自动校验（生成前必做）

生成前必须检查并修正以下问题：
- 时间重叠：同一时段不得安排两个任务。
- 固定日程覆盖：已确定事项不得被深度工作块覆盖。
- 总时长超载：计划总时长不得超过可用工时。
- 连续工作过长：连续深度工作超过 120 分钟时，自动插入休息或缓冲。
- 午间空档：若跨中午连续排满，建议补充午休/恢复时段（可由用户覆盖）。

## 输出文档（Obsidian）

### 生成规则
- 保留用户原意，不擅自改目标。
- 对明显冲突做最小修正（例如时间重叠），并在文末“备注”说明。
- 重点工作默认不超过 3 项；超出时建议用户降优先级。
- 子任务优先写成“动词开头”的动作项。
- 若用户未给日期，先询问日期后再生成。
- 若用户说“随便你决定/你判断/按你来”，自动判定模式并在 frontmatter 写入 `mode`：
  - `mode: compact`：时间紧、信息缺失较多、用户强调快速产出、可用时段较少（如 <= 3 小时）。
  - `mode: standard`：目标较复杂、截止压力高、需要多重点并行、用户希望完整规划。
- 自动判定仅为默认策略；用户显式指定模式时必须覆盖自动判定结果。
- 若 `daily_plan/outputs/` 已存在当天文件，优先更新该文件而非创建重复文件。
- 成稿必须写入 `daily_plan/outputs/YYYY-MM-DD-daily-plan.md`。

### 精简模式分支（用户明确要求时启用）

触发条件（任一满足即启用）：
- 用户明确说“精简模式/简版/轻量版/最小版”
- 用户明确要求“少一点、只保留关键、快速生成”

执行规则：
- 仍遵循三阶段主流程（允许按既有规则简化阶段 A）。
- 问题收集压缩为 3-4 问：
  1) 今日日期 + 今日 Top1  
  2) 固定日程（含时间）  
  3) 至少一个专注时段  
  4) 是否并入递延任务（可选）
- 重点工作最多 2 项，子任务每项 1-3 个。
- 时间块最少 2 块，最多 6 块。
- 仍需执行容量控制与时间冲突校验，不因精简而跳过。

## 晚间复盘闭环（可选但建议）

当用户在晚间复盘时，追加 3 问并更新当日复盘文件（任务完成勾选以当日日计划文件为准，不要求在复盘中逐条勾选）：
- 今天完成了什么（Done）？
- 主要卡点是什么（Blockers）？
- 明天第一优先是什么（Top1 for tomorrow）？

若用户同意，自动生成次日候选任务列表并写入：
- `daily_plan/archive/YYYY-MM-DD-retro-notes.md`（可选）

## 质量检查清单（生成前自检）

输出前确认：
- 是否遵循“三阶段”流程（若简化，是否已记录简化原因）
- 是否覆盖输出模板全部板块（含递延记录与备注）
- 是否可直接粘贴到 Obsidian 使用
- 是否包含可勾选任务与时间块
- 是否避免空泛语句，保留用户原始目标
- 是否已写入 `daily_plan/` 约定路径
- 是否进行了递延任务三分法处理
- 是否做了容量控制并处理超载
- 是否完成时间冲突自动校验（重叠/覆盖/超载/连续时长）
- 若为精简模式，是否使用精简模板并满足最小字段
- 若用户未指定模式且触发自动判定，是否已写入正确的 `mode` 并与输出结构一致
